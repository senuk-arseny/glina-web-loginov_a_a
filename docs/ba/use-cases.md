# use-cases.md — Гончарная мастерская «Глина»

Каждый UC содержит `Realizes` — обратную ссылку на user story(и), которую он реализует (см. `user-stories.md`).

## UC-1: Бронирование занятия
**Realizes:** US-01, US-02, US-03, US-04

**Preconditions:** клиент вошёл в приложение; на слот есть свободные места.

**Main flow:**
1. Клиент открывает список слотов на ближайшие 7 дней. *(US-01)*
2. Клиент выбирает слот, открывает детали. *(US-02)*
3. Клиент нажимает «Записаться», опционально выбирает прокат. *(US-03, US-04)*
4. Бэкенд (в MVP — мок-слой) атомарно проверяет наличие места, лимит программы и отсутствие дублирующей брони этого клиента.
5. Бронь подтверждается, клиент видит подтверждение, отправляется уведомление.

**Alternative/Error flow:**
- 4a. Мест не осталось → «Мест больше нет» + предложение листа ожидания.
- 4b. Клиент уже записан на этот слот → отказ «Вы уже записаны на это занятие» (US-03 AC).
- 3a. Прокат недоступен → опция неактивна.

**Postconditions:** создана Booking (active), отправлено уведомление.

## UC-2: Отмена записи
**Realizes:** US-05

**Preconditions:** у клиента есть активная бронь на будущий слот.

**Main flow:**
1. Клиент открывает историю записей, нажимает «Отменить».
2. Бэкенд определяет тип отмены (ранняя ≥2ч / поздняя <2ч) и возвращает результат.
3. Клиент видит подтверждение отмены; при поздней — предупреждение.
4. Прокат (если был) возвращается в доступный фонд.
5. Если есть лист ожидания — первый в очереди уведомляется.

**Postconditions:** Booking → cancelled_by_client; при поздней отмене — счётчик в профиле; waitlist продвинут.

---

## UC-3: Лист ожидания
**Realizes:** US-06

**Preconditions:** слот полностью заполнен и не отменён мастерской.

**Main flow:**
1. Клиент видит кнопку «Встать в лист ожидания» вместо «Записаться».
2. Создаётся WaitlistEntry, клиент видит свою позицию в очереди.
3. При отмене чьей-то брони — первый в очереди уведомляется, статус → notified.
4. Клиент бронирует освободившееся место обычным способом (UC-1).

**Alternative/Error flow:**
- 1a. Слот отменён мастерской → запись в лист ожидания недоступна (см. UC-6).

**Postconditions:** WaitlistEntry создана/обновлена; уведомление отправлено.

---

## UC-4: Оценка мастера
**Realizes:** US-07

**Preconditions:** у клиента есть завершённая бронь без оценки.

**Main flow:**
1. Клиент открывает завершённую запись, нажимает «Оценить мастера».
2. Ставит рейтинг (1-5) и опционально комментарий.
3. Отзыв сохраняется, средний рейтинг мастера пересчитывается.
4. Клиент видит подтверждение отправки отзыва.

**Alternative/Error flow:**
- Повторная попытка оценить ту же бронь → блокируется.

**Postconditions:** создан Review, обновлён avgRating мастера.

## UC-5: Вход по имени и телефону
**Realizes:** US-11

**Preconditions:** нет.

**Main flow:**
1. Клиент открывает главную страницу, нажимает «Войти в мастерскую».
2. Вводит имя и телефон.
3. Система нормализует телефон и ищет клиента по нему; если не найден — создаёт нового.
4. Клиент попадает в кабинет (список занятий).

**Postconditions:** создан или обновлён ClientProfile, сессия активна.

## UC-6: Форс-мажорная отмена занятия мастерской
**Realizes:** US-08

*(Ранее отсутствовал в документе, хотя US-08, FR-11, FR-14 и вся фича `feature-03-cancellation.md` уже существовали — закрывает пробел.)*

**Preconditions:** у клиента есть активная бронь на слот, который отменяет мастерская (форс-мажор — R-008).

**Main flow:**
1. Мастерская отменяет занятие в своей инфраструктуре (само действие — вне скоупа клиентского приложения).
2. Бэкенд переводит статус слота в `cancelled_by_studio` и статус всех активных броней на этот слот — тоже в `cancelled_by_studio`.
3. Клиент получает уведомление с указанием причины отмены.
4. При открытии деталей слота или списка «Мои записи» клиент видит статус «Отменено мастерской» с причиной.

**Alternative/Error flow:**
- Клиенты, стоявшие в листе ожидания на этот слот, теряют место в очереди (статус → `expired`); повторная запись на слот запрещена, в том числе через лист ожидания (см. UC-3, альтернативный поток).

**Postconditions:** Booking.status = `cancelled_by_studio` для затронутых броней; WaitlistEntry.status = `expired` для затронутых записей; уведомление отправлено каждому затронутому клиенту.

## UC-7: Просмотр уведомлений
**Realizes:** US-09

*(Ранее отсутствовал — уведомления упоминались только как побочный эффект внутри UC-1/UC-2/UC-3, без собственного сценария просмотра.)*

**Preconditions:** клиент вошёл в приложение.

**Main flow:**
1. Клиент открывает раздел «Уведомления».
2. Система проверяет, не пора ли создать напоминания по активным бронированиям (до занятия осталось меньше 24 часов), и добавляет их в список.
3. Клиент видит список уведомлений, отсортированный от новых к старым.
4. Открытие раздела помечает все уведомления клиента прочитанными; счётчик непрочитанных в навигации сбрасывается.

**Alternative/Error flow:**
- 3a. Уведомлений нет → пустое состояние с предложением перейти к расписанию.

**Postconditions:** Notification.read = true для всех уведомлений клиента.

---

## UC-8: Просмотр профиля
**Realizes:** US-10

*(Ранее отсутствовал — несмотря на отдельный экран профиля с прогрессом до статуса.)*

**Preconditions:** клиент вошёл в приложение.

**Main flow:**
1. Клиент открывает раздел «Профиль».
2. Система показывает имя, контакт, число завершённых посещений, число поздних отмен.
3. Если статус «постоянный клиент» ещё не достигнут — показывается прогресс (сколько посещений осталось до 3).
4. Если достигнут — показывается бейдж «Постоянный клиент» вместо прогресса.

**Postconditions:** нет (сценарий только для чтения).
